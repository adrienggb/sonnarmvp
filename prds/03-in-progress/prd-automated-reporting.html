Décomposition du problème

Pain A — Report client hebdo (~6-7h)

Aujourd'hui : Le consultant ouvre Airtable, compte manuellement les candidats à chaque stage du funnel, screenshot la table de performance, ouvre Gmail, rédige un mail avec les KPIs, les candidats rencontrés, les actions prévues. Envoie.

Problèmes : KPIs self-reported (pas system-derived), format inconsistant entre consultants, data en retard d'une semaine, aucune visibilité client entre deux envois, screenshot non interactif.

Pain B — CR d'entretien (~3-4h)

Aujourd'hui : Le consultant enregistre le call (audio), passe la transcription dans ChatGPT/Gemini, copie le résultat, reformate manuellement, colle dans Airtable (champ CR du qualification record).

Problèmes : Pas de prompt standardisé → qualité variable. Pas de structure imposée → certains CR font 3 lignes, d'autres 3 pages. Pas de lien entre le CR et les champs structurés (score, forces, faiblesses).

Solution — Vue d'ensemble
2 modules, 1 stack
Module A — Client Report Auto : Un portail client (page web statique par mission) qui se met à jour en temps réel depuis Airtable. + Un email hebdo auto-généré par Claude (KPIs + résumé + actions) envoyé via Gmail/n8n. Le consultant ne fait plus que relire et valider avant envoi.

Module B — CR d'entretien structuré : Un workflow n8n qui prend la transcription audio (Metaview, Grain, ou upload manuel), la passe dans Claude API avec un prompt structuré, et injecte le CR formaté + les champs structurés directement dans Airtable. Le consultant ne fait plus que relire et ajuster le score.

Détail du flux

1. Collecte (n8n) : Le workflow interroge Airtable via API pour la mission concernée. Il récupère : tous les candidats du pipe avec leur stage actuel, les mouvements de la semaine (nouveaux candidats, changements de stage), les qualification records créés/modifiés cette semaine, les KPIs du funnel (comptés automatiquement par stage, pas self-reported).

2. Génération (Claude API) : Les données sont envoyées à Claude avec un prompt structuré qui produit : un résumé exécutif (3-4 phrases), un tableau de KPIs formaté, la liste des candidats rencontrés cette semaine (avec résumé 2 lignes de chaque), les actions prévues pour la semaine suivante, un signal d'alerte si un KPI dévie des benchmarks.

3. Livraison (Gmail) : Le résultat est injecté dans un draft Gmail dans la boîte du consultant (pas envoyé directement). Le consultant relit, ajuste le ton si besoin, et envoie. Ceci préserve le contrôle humain et la relation client.

4. Portail (Airtable Interface) : En parallèle, une vue Airtable Interface (ou page HTML) est partagée au client via un lien. Elle affiche le funnel live, les candidats présentés, le timeline. Mise à jour automatique — le client peut consulter à tout moment sans attendre l'email.

Détail du flux

1. Input : V1 (MVP) : le consultant colle la transcription dans un formulaire Airtable ou une interface simple (+ sélectionne mission et candidat). V2 : Metaview envoie la transcription automatiquement via webhook à n8n.

2. Structuration (Claude API) : Le prompt reçoit la transcription + les critères d'évaluation de la mission (récupérés depuis Airtable). Il produit : un CR structuré (contexte, parcours, motivations, forces, faiblesses, recommandation), un score suggéré (1-5), les champs structurés extraits (nombre de personnes managées, salaire actuel/cible, localisation souhaitée, disponibilité, aspirations).

3. Injection (Airtable API) : Le CR est écrit dans le qualification record. Les champs structurés sont mis à jour dans le candidate record. Le score est marqué comme "suggéré par IA" — le consultant peut override.

4. Feedback loop : Si le consultant modifie le score ou le CR, cette correction est loguée. À terme, ces corrections alimentent l'amélioration du prompt (mais hors scope MVP).

Point d'attention : KPIs system-derived vs self-reported
Aujourd'hui les KPIs du funnel sont saisis manuellement par le consultant. Pour que le Module A fonctionne, il faut que les KPIs soient dérivés automatiquement des stages du pipeline dans Airtable. Cela signifie que le champ pipeline_stage doit être un enum strict (pas du texte libre) et que chaque changement de stage doit être timestampé. C'est un prérequis bloquant — tâche T0 du projet.

Edge cases, risques d'adoption, et considérations non évidentes.
Adoption & Change Management

🧠 Les consultants ne font pas "que" du reporting

Les 10h incluent aussi du temps de réflexion stratégique que le consultant utilise pour prendre du recul sur la mission. Le mail hebdo est aussi un moment où le consultant structure sa pensée. Si on automatise le mail, on risque de perdre ce moment de prise de recul.

Mitigation : Le workflow génère un draft, pas un envoi. Le consultant relit, et le prompt inclut une section "actions recommandées" qui force la réflexion stratégique. On transforme le reporting en review assistée plutôt qu'en tâche supprimée.

🔒 Confidentialité des données candidats

Les transcriptions d'entretiens contiennent des données personnelles sensibles (salaire, situation familiale, motivations confidentielles). Les envoyer à Claude API signifie qu'elles transitent par un tiers.

Mitigation : Claude API en mode zero data retention. Vérifier la conformité RGPD de l'usage. Anonymiser les transcriptions avant envoi si nécessaire (retirer les noms d'entreprise, numéros de tel). Documenter le flux dans le registre de traitement RGPD de Sonnar.

📧 Le client répond au mail auto-généré

Aujourd'hui le mail est écrit par le consultant → le client répond au consultant. Si le mail est auto-généré, le ton peut ne pas matcher la relation. Le client peut aussi poser une question sur un détail du report que le consultant n'a pas relu en profondeur.

Mitigation : Le mail est envoyé DEPUIS la boîte du consultant (pas un no-reply). Le draft permet au consultant d'ajuster le ton. Le prompt doit capter le contexte relationnel (nouveau client vs relation longue). V2 : stocker le ton des précédents mails pour consistency.

📊 Missions sans mouvement cette semaine

Certaines semaines, rien ne bouge sur une mission (consultants en congé, client en décision, marché sec). Que fait-on ? Un mail vide ? Pas de mail ?

Mitigation : Le prompt détecte l'absence de mouvement et génère un message adapté ("Cette semaine, nous avons consolidé les candidatures en cours. Voici le plan pour la semaine prochaine..."). Option : le consultant peut skip l'envoi via un bouton dans Slack ("Pas de report cette semaine, mission en pause").

Edge cases techniques

⚠️ Rate limits Airtable Free plan

Airtable Free : 5 API requests/sec, 1000 records/base. Si on a 17 consultants × 6 missions = 102 missions actives, et le cron du lundi fetch toutes les missions + pipeline data + CR → on peut dépasser les limites.

Mitigation : Stagger les appels dans n8n (traiter les missions séquentiellement avec un délai de 2 sec entre chaque). Si Free plan ne suffit pas, Team plan (~20$/user/mois) donne 100K records et des limites API plus hautes. À budgéter.

⚠️ La migration Hiresweet en cours

Si l'ATS migre vers Hiresweet pendant qu'on construit le MVP sur Airtable, les sources de données changent. Le pipeline vit peut-être dans Hiresweet, les candidats dans Airtable, ou un mix des deux.

Mitigation : Abstraire la source de données dans n8n. Un node "fetch_mission_pipeline" qui peut pointer vers Airtable OU Hiresweet. Commencer sur Airtable (existant), mais coder le workflow avec cette flexibilité en tête. Clarifier avec Cyrille/Thibaut : quelle est la timeline de migration ?

⚠️ Multi-ATS client

Certains clients demandent que les candidats soient aussi entrés dans leur propre ATS (TeamTailor, Lever...). Le portail Sonnar ne remplace pas cette exigence. Le consultant devra potentiellement toujours faire la saisie dans l'ATS client.

Mitigation : Le portail Sonnar est un complément, pas un remplacement de l'ATS client. Mais il réduit la pression sur le reporting email. V2 : explorer des intégrations push vers les ATS clients les plus courants (API TeamTailor, API Lever).

Impact positif non évident

💡 Résout partiellement le problème #3

En extrayant automatiquement des champs structurés depuis les CR (nombre de direct reports, salaire, dispo, aspirations), le Module B crée de la donnée recherchable. C'est une fondation pour la recherche sémantique du problème #3 — sans avoir à construire un pipeline NLP complet maintenant.

💡 Alimente directement le levier 40 k€

Un portail client live + des reports auto-générés de qualité consistent = un argument tangible pour justifier le positionnement premium. C'est un sales asset, pas juste un outil ops. Cyrille peut dire en call client : "Vous aurez une visibilité temps réel sur votre mission via votre portail dédié."

💡 Les KPIs system-derived débloquent le capacity model

Une fois les KPIs auto-calculés (Phase 0), on a les données pour construire le sync Sales↔Delivery identifié comme risque critique. On sait en temps réel combien de missions chaque consultant gère, à quel stage elles sont, et quand elles se terminent. C'est la base du capacity dashboard pour éviter l'overselling.
